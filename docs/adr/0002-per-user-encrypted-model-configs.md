# Per-user encrypted model configs with YAML-based catalog

Users should be able to bring their own API keys for any ai-sdk-compatible provider without the app admin needing to pre-configure every permutation. A YAML catalog at `src/config/models.yaml` declares the supported providers (mapped to ai-sdk packages/constructors) and their model flavors — this is the single source of truth for what users can pick. Each user stores their own provider+model+encrypted API key in a `user_models` table; the active selection is a FK on the `user` table. Encryption is done at the DB level via pgcrypto to keep keys out of application memory whenever possible. System-level tasks (title generation, etc.) remain on separate env-var models, invisible to users. Considered alternatives included all-env-var setup (limits user choice, burdens admin) and app-level Node crypto (works but pgcrypto keeps the key material in the DB layer where it's harder to leak via log statements or error handling).
